# Step 06 â€” Worker Audit Repository

## Scope

**In scope:**

- Synchronous repository module for writing audit events and reference audit entries from the worker
- `JobEventType` enum for the worker
- Batch insert for reference audit entries
- Helper to build reference audit entries from `ResultsV1` data
- Unit tests

**Out of scope:**

- Integrating the repo into the pipeline runner and stages (Step 07)
- Reading audit data from the worker

## Context

The worker accesses Supabase synchronously via `apps/worker/biblio_checker_worker/jobs/repo.py`. That module receives `supabase: Client` as an explicit parameter, uses `_sanitize_detail()` for error_detail truncation, and raises `JobRepoError` on failures.

The audit repo follows the same synchronous client pattern but with fire-and-forget semantics (no exceptions propagated).

The worker is responsible for emitting all event types except `job_created` (which is backend-only), and for writing reference audit entries after successful verification.

### R3 â€” `insert_job_event` function

```python
def insert_job_event(
    supabase: Client,
    *,
    job_id: str,
    event_type: str,
    stage: str | None = None,
    status: str | None = None,
    error_code: str | None = None,
    error_detail: str | None = None,
    attempt: int | None = None,
    metadata: dict | None = None,
) -> None:
```

**Behavior:**

1. Build a row dict from parameters (use `{}` if `metadata` is None)
2. Truncate `error_detail` to 200 characters
3. Call `supabase.table("job_events").insert(row).execute()`
4. On ANY exception: log at WARNING level, return normally
5. Never re-raise â€” fire-and-forget

### R4 â€” `build_reference_audit_entry` function

```python
def build_reference_audit_entry(
    *,
    job_id: str,
    reference_id: str,
    raw_text: str | None,
    classification: str | None,
    confidence_score: float | None,
    reason_code: str | None,
    sources_checked: list[str],
    match_found: bool | None,
    error_detail: str | None = None,
) -> dict:
```

**Behavior:**

1. Pure function â€” no I/O, no side effects
2. Returns a dict with keys matching the `reference_audit_log` column names
3. Includes `job_id` in the returned dict
4. Truncates `error_detail` to 200 characters
5. Does NOT include `id` or `created_at` (auto-generated by DB)

**Return value example:**

```python
{
    "job_id": "abc-123",
    "reference_id": "ref-001",
    "raw_text": "Smith et al. (2024) ...",
    "classification": "suspicious",
    "confidence_score": 0.85,
    "reason_code": "title_fuzzy_only",
    "sources_checked": ["openalex", "scielo", "arxiv"],
    "match_found": True,
    "error_detail": None,
}
```

### R5 â€” `insert_reference_audit_batch` function

```python
def insert_reference_audit_batch(
    supabase: Client,
    *,
    job_id: str,
    entries: list[dict],
) -> None:
```

**Behavior:**

1. If `entries` is empty, return immediately (no DB call)
2. Call `supabase.table("reference_audit_log").insert(entries).execute()` â€” single bulk insert
3. On ANY exception: log at WARNING level with `job_id` and count of entries, return normally
4. Never re-raise â€” fire-and-forget

**Logging format:**

```
WARNING audit_repo: Failed to insert reference audit batch [job_id=%s, count=%d]: %s
```

## Edge Cases

| Scenario | Expected Behavior |
|---|---|
| `entries` list has 200+ dicts | Single bulk insert; Supabase client handles it |
| `entries` list is empty | Return immediately, no DB call |
| Supabase client is None or invalid | Exception caught, logged as WARNING |
| `build_reference_audit_entry` receives None for all optional fields | Returns dict with None values; valid for DB insertion |
| `sources_checked` is an empty list | Stored as `'{}'` in Postgres |
