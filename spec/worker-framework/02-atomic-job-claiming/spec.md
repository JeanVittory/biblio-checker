# Step 02 — Atomic Job Claiming

## Scope

- Define the Postgres RPC function `claim_analysis_job` that atomically selects and claims a job
- Define the lease model (token + TTL) for crash recovery
- Define the concurrency guarantees
- Define the migration to be applied to the Supabase project

**Out of scope:** Python code that calls the RPC (see Step 04). Retry strategy (see Step 10).

## Context

The worker uses Supabase's Python client, which translates all operations to PostgREST HTTP calls. PostgREST does not expose `SELECT ... FOR UPDATE SKIP LOCKED`, making it impossible to atomically select-and-update a row from application code. A naive `SELECT` followed by `UPDATE` has a TOCTOU (Time-of-Check-Time-of-Use) race condition: two workers can both read the same queued row before either writes.

The `analysis_jobs` table already has two columns designed for this purpose:
- `job_token` (text, nullable): a secret token written at claim time
- `token_expires_at` (timestamptz, nullable): lease expiry timestamp

The backend writes an initial `job_token` with a 1-hour TTL when creating the job (for frontend status polling authentication). The worker must overwrite these with its own short-lived lease token at claim time.

## Requirements

### 1. RPC Function Specification

**Name:** `public.claim_analysis_job`

**Parameters:**

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `p_token` | text | (required) | Unique lease token generated by the worker |
| `p_lease_secs` | int | 300 | Lease duration in seconds (5 minutes) |

**Return type:** `SETOF analysis_jobs` (0 rows if no job available, 1 row if a job was claimed)

**Language:** PL/pgSQL

**Security:** `SECURITY DEFINER` (runs as function owner, bypassing RLS)

### 2. Claiming Logic

The function MUST perform a single atomic operation that:

1. Selects one eligible job using these criteria:
   - `status = 'queued'`
   - `attempts < max_attempts` (exhausted jobs are excluded)
   - `token_expires_at IS NULL OR token_expires_at < now()` (allows reclaiming jobs with expired leases)
2. Orders candidates by `created_at ASC` (FIFO — oldest job first)
3. Applies `FOR UPDATE SKIP LOCKED` to avoid blocking concurrent workers
4. Updates the selected row atomically:
   - `status` = `'running'`
   - `stage` = `'created'`
   - `job_token` = `p_token`
   - `token_expires_at` = `now() + p_lease_secs`
   - `attempts` = `attempts + 1`
   - `updated_at` = `now()`
5. Returns the updated row via `RETURNING *`

### 3. Concurrency Guarantees

- **No double-claiming:** `FOR UPDATE SKIP LOCKED` ensures that if worker A is in the process of claiming a row, worker B skips it and moves to the next eligible row (or returns empty).
- **No deadlocks:** `SKIP LOCKED` never waits for a lock; it simply skips contested rows.
- **FIFO ordering:** `ORDER BY created_at ASC` processes the oldest queued jobs first. The `SKIP LOCKED` clause may cause slight deviations from strict FIFO under high contention, but this is acceptable.
- **Single-row claim:** `LIMIT 1` ensures each RPC call claims at most one job. The worker processes one job per poll cycle.

### 4. Lease Model

- Each claim writes a unique `job_token` (generated by the worker as `secrets.token_urlsafe(32)`) and a `token_expires_at` timestamp.
- The lease duration defaults to 300 seconds (5 minutes), configurable via `p_lease_secs`.
- If the worker completes or fails the job, it clears the token and expiry (see Step 04).
- If the worker crashes without clearing the token, the lease expires naturally. The next poll cycle by any worker will reclaim the job because `token_expires_at < now()` becomes true.

### 5. Attempt Tracking

- `attempts` is incremented at claim time, not at failure time. This means a crash (where the worker never reports back) still counts as an attempt.
- Jobs with `attempts >= max_attempts` are never returned by the RPC. They remain in their current state until manual intervention.

### 6. Permission Model

- `EXECUTE` is revoked from `PUBLIC`
- `EXECUTE` is granted only to `service_role`
- The worker connects using the `SUPABASE_SERVICE_ROLE_KEY`, which authenticates as `service_role`

## Acceptance Criteria

- The RPC function is created successfully in the Supabase project
- Calling the RPC with a valid token returns exactly one job row when queued jobs exist
- Calling the RPC returns zero rows when no eligible jobs exist
- The returned row has `status = 'running'`, `stage = 'created'`, the provided token, correct `token_expires_at`, and `attempts` incremented by 1
- Two concurrent RPC calls with one queued job result in exactly one claim (the other returns empty)
- A job with `attempts = max_attempts` is never returned
- A job with an expired `token_expires_at` is eligible for re-claiming
- A job with a non-expired `token_expires_at` is not eligible for claiming (another worker holds the lease)
- The RPC is callable only by `service_role`, not by `anon` or `authenticated`

## Edge Cases

| Scenario | Expected Behavior |
|----------|-------------------|
| No queued jobs in the table | RPC returns empty set (0 rows) |
| Multiple queued jobs | The oldest by `created_at` is claimed first |
| Job has `token_expires_at` in the future (another worker holds it) | Job is skipped; not eligible for claiming |
| Job has `token_expires_at` in the past (previous worker crashed) | Job IS eligible; the RPC re-claims it |
| Job has `token_expires_at = NULL` (initial state from backend) | Job IS eligible; NULL is treated as "no lease" |
| Job has `attempts = 2` and `max_attempts = 3` | Eligible (2 < 3); this will be the last attempt |
| Job has `attempts = 3` and `max_attempts = 3` | NOT eligible (3 < 3 is false); permanently excluded |
| 10 workers call the RPC simultaneously with 1 queued job | Exactly 1 worker gets the job; 9 get empty results |
| Worker calls RPC with `p_lease_secs = 0` | Token expires immediately; job could be reclaimed on next poll. Technically valid but not recommended. |
